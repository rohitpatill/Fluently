# backendAppRoutersContext.md — scope: `backend/app/routers/`

HTTP layer only — validation + delegation to services/`repo`. Sync `def` routes (FastAPI threadpools them).
Parent: [../backendAppContext.md](../backendAppContext.md).
Routes call `repo` / services directly (no `db` session). Every data route resolves the real user via `user_id: str = Depends(get_current_user)` (from `app/deps.py`) — no more hardcoded `DEFAULT_USER_ID`; a missing/invalid session ⇒ 401. **LLM-using routes** (chat send, conversation create, opener) instead depend on `Depends(require_model_configured)` → 403 "No model configured" if the user has no key/tier yet (frontend turns this into the model-config gate). Path ids are strings (ObjectId hex); an invalid/absent id yields 404. `voice.py` is the one **WebSocket** router (real-time audio) — it authenticates the session cookie itself on the handshake (the HTTP `Depends` chain can't raise on a WS).

| File | Endpoints |
|---|---|
| `auth.py` | Google OAuth + session (prefix `/api/auth`). `GET /google/login` (sets a signed HttpOnly state+nonce cookie, 307 → Google consent). `GET /google/callback` (verify state/CSRF → exchange code → verify ID token → `upsert_user_from_google`; first-ever user adopts `default` data; bootstraps the user's memory files; sets the JWT session cookie; redirects to `frontend_url`; any failure → `frontend_url/?auth_error=1`, no session set). `GET /me` (current profile + `has_persona` + **`has_key`/`tier`**; 401 if unauthenticated). `POST /logout` (clears the session cookie). All state/session set+delete_cookie calls use `settings.cookie_secure` + `settings.cookie_samesite` (→ `SameSite=None; Secure` when frontend/backend are cross-domain, else `Lax`) so the session survives a split deploy; delete uses the same flags or the cookie won't clear. |
| `model.py` | Bring-your-own-key model config (prefix `/api/model`). `GET /tiers` (Swift/Sage catalogue — name/model id/tagline/price, no secrets). `GET /status` (`{has_key, tier}` for the current user — NEVER the key). `POST /key` (`{api_key, tier}` → `model_service.verify_key` → `crypto_service.encrypt` → `repo.set_user_key`; 400 if verify fails / unknown tier / empty). `PUT /tier` (`{tier}` → switch with the already-stored key; 400 if unknown tier or no key). Powers both onboarding step 3 and Settings' Brain card. |
| `chat.py` | `POST /api/chat/{conversation_id}` — full chat turn; returns user_message, assistant_message (with tool_calls), scoring_events. |
| `personas.py` | Multi-persona management (prefix `/api/personas`). `GET ""` (list, each with active flag + conversation_count), `GET /catalog` (curated Discover catalog grouped by category, from `persona_catalog`) + `POST /catalog/{catalog_id}/use` (COPY a catalog figure — incl. its hand-picked `voice` — into the user's personas; does NOT switch active), `POST ""` (create), `PUT /{id}` (edit identity fields incl. `gender`/`voice`, preserves relationship memories + avatar), `PUT /{id}/avatar` (set/clear public URL), `POST /{id}/activate` (switch active persona), `DELETE /{id}` (delete persona + its conversations/messages; keeps words; refuses the LAST one; auto-switches active if the deleted one was active). Static `/catalog` routes are declared BEFORE `/{persona_id}` so they win. |
| `conversations.py` | `POST /api/conversations` (picks target words + stamps the ACTIVE persona's id via `memory_service.active_persona_id`, optional topic suggestions scoped to that persona), `GET` list (**persona-scoped — only the active persona's threads**)/detail, `GET .../{id}/messages` (each message carries `tool_calls`; USER messages also carry `word_events` — that message's scoring events with resolved `word_text` — so scoring chips + Developer-mode tool calls survive a refresh), `PATCH .../category` (query param), `POST .../opener` (400 if messages exist), `DELETE`, `POST /api/conversations/search` (BM25/regex, window flags; **scoped to the active persona**). |
| `words.py` | `GET /api/words` (applies lazy decay), `POST` (409 duplicate, LLM enrichment on add), `GET/DELETE /{id}`, `PUT /{id}/note` (user's own memory-hook note; trims, empty clears), `POST /{id}/adjust` (manual event), `GET /{id}/events` (last 100). |
| `memory.py` | `GET /api/memory/{file}` (raw + parsed lines, no ids/timestamps), `POST .../lines` (append), `POST .../edit` (old_string→new_string, empty new_string deletes; 404 if old_string absent, 400 if empty), `PUT .../{file}/raw` (whole-file overwrite, 400 without "raw" key), `PUT /api/memory/persona/form` (onboarding step 1), `POST /api/memory/onboarding` (onboarding step 2 finish: stores Name deterministically + LLM-structures the free-text "about" box across identity/memory/persona via `topic_service.structure_onboarding_info`; falls back to raw-append to identity on LLM failure). Valid files: identity, memory, persona. |
| `voice.py` | Voice mode — real-time duplex audio via Gemini Live (prefix `/api/voice`). **`WS /ws/{conversation_id}`**: cookie-auth on the handshake (`_authenticate`, NOT the HTTP dep — WS can't raise HTTPException), then proxies browser↔Gemini through `voice_service.open_session` — two relay coroutines (`_browser_to_gemini` mic 16k PCM in; `_gemini_to_browser` audio 24k PCM + transcripts + tool calls out). The receive loop is `while True: async for … in session.receive()` (receive() ends each turn — the outer loop re-enters it, else the session dies after turn 1). Executes tool calls SERVER-SIDE (`VoiceToolExecutor` → `send_tool_response`); a `score_word` result is pushed as a `{"type":"score"}` frame (live animation). `_TurnBuffer` persists EACH completed turn as normal user+assistant `Message`s (assistant carries `tool_calls`; live score events get their `message_id` back-filled to the user msg via `repo.set_event_message_id`) so the voice chat shows in the thread afterward. **REST:** `GET /voices` (voice catalogue for the persona-form picker + `audition_url` to Google AI Studio), `GET /status` (`{available}` = user has a key/tier). |
| `dashboard.py` | `GET /api/dashboard/stats` — totals, mastered, average, weekly events/gains, top/weakest/slipping words (via `repo` aggregations). |
| `settings.py` | Data-management HARD deletes: `DELETE /api/settings/conversations` (purge conversations+messages, keep words/memories), `DELETE /api/settings/memories` (reset identity+memory only — the shared human files; personas untouched), `POST /api/settings/purge-all` (`{keep_words}` — nuke conversations + **ALL personas** (`repo.purge_personas`) + reset identity/memory, optionally words+events; both flows restart at onboarding which recreates one persona; **when `keep_words=False` it's a TRUE full reset that ALSO clears the user's model key+tier via `repo.clear_user_model`** → re-onboard the brain; response includes `cleared_model`). All via `repo.purge_*`/`clear_user_model` + `memory_service.reset_file`. |

Conventions: raise `HTTPException` with clear detail; response models from `schemas.py`; no business logic here.
