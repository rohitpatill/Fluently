# backendAppServicesContext.md — scope: `backend/app/services/`

All business logic. Parent: [../backendAppContext.md](../backendAppContext.md).

| File | Purpose & key functions |
|---|---|
| `auth_service.py` | Google OAuth + JWT session — pure, framework-agnostic helpers (no FastAPI types; unit-testable). `new_state_and_nonce`, `sign_state`/`unsign_state` (itsdangerous `TimestampSigner`, 300s TTL — the stateless CSRF/replay guard), `build_google_auth_url` (scope `openid email profile`, `prompt=select_account`), `exchange_code_for_tokens` (httpx POST to Google's token endpoint), `verify_google_id_token` (`google-auth` checks signature/exp/aud; we ALSO assert `iss`, `nonce`, `email_verified`), `mint_session_jwt`/`decode_session_jwt` (PyJWT HS256, algorithm PINNED). `AuthError` on any failure. Stores NO Google tokens — reads identity from the ID token and discards. |
| `llm_service.py` | Model factory via LangChain `init_chat_model`, now **per-user key-aware**. `get_chat_model(provider, model, api_key=None, **kw)`, `get_judge_model(...)` (temp 0 default), `get_utility_model(...)` — `api_key` is passed straight into `init_chat_model` (NO `os.environ` mutation, so concurrent users' keys never clobber). Providers: openai / anthropic / google_genai (Gemini MUST use explicit `google_genai`). |
| `crypto_service.py` | Fernet symmetric encryption for users' bring-your-own API keys. `encrypt(plaintext)->ciphertext`, `decrypt(ciphertext)->plaintext`; `CryptoError` on misconfig/tamper. Key from `settings.encryption_key` (`.env` `ENCRYPTION_KEY`, generate via `Fernet.generate_key()`) — NEVER stored in the DB. The single seam that does key crypto. |
| `model_service.py` | Per-user model resolution. `tiers_public()` (Swift/Sage catalogue for the UI, no secrets), `resolve_for_user(user_id)->ResolvedModel{provider,model,api_key,tier}` (loads user, decrypts key, maps tier→model id; raises `NoModelConfigured` if unset), `verify_key(api_key, tier)->bool` (one throwaway `invoke("hi")` with `max_output_tokens=1`, never raises). Tier→model map lives in `config.MODEL_TIERS` (swift=gemini-3.1-flash-lite, sage=gemini-3.5-flash). |
| `voice_service.py` | Gemini Live (real-time duplex audio) session wrapper — the ONLY module that knows the live-model config. `open_session(conversation)` async CM yields `(session, voice)` via `client.aio.live.connect(model=settings.voice_model)` using the user's OWN decrypted key (`resolve_for_user`). `build_live_config` = audio-out + locked `voice` + **the SAME text system prompt (`prompt_builder.build_system_prompt`) with `prompts.VOICE_MODE_INSTRUCTION` appended** (voice inherits all context; override makes it score via the tool + speak conversationally) + tools (`voice_tools`) + in/out transcription. `voice_for_conversation` = the active persona's `Voice:` field, defaulted from Gender (`config.resolve_voice`). |
| `assistant_service.py` | The Fluently in-app VOICE ASSISTANT (the app talking to the user for help + hands-free actions) — SEPARATE from persona voice chat: nothing persisted, nothing scored, no persona. `build_assistant_prompt(user_id, current_tab)` fills `prompts.ASSISTANT_SYSTEM_TEMPLATE` (explains every feature + a hard STAY-IN-SCOPE boundary) with the user's name + identity+memory (loaded so it sounds custom) + active persona name + which app screen they're on. `get_status(user_id)` = the live numbers for the `get_my_status` tool (word counts/avg/weakest, persona+conversation counts, days-using, tier). `build_live_config(...)` = the Gemini LiveConnectConfig (audio out, a FIXED friendly voice `Puck` — the app's own voice, NOT the persona's — the assistant prompt, assistant tool declarations, transcription). Reuses ONLY the transport + per-user key. |
| `assistant_tools.py` | The assistant's Live tools (adapts app actions to the Live protocol, like `voice_tools` but for help/setup). `assistant_function_declarations()` = FOUR tools: `get_my_status` (read-only), `create_persona`, `add_word`, `switch_model_tier`. `AssistantToolExecutor.execute(name,args)` runs each server-side by WRAPPING existing code paths (create_persona → `memory_service.build_persona_content`+`repo.insert_persona`; add_word → `topic_service.enrich_word`+`repo.insert_word`, same as POST /api/words; switch_model_tier → `repo.set_user_tier`), records each call, returns a JSON-able result (with `action`+`message` on success → router pushes an on-screen confirmation). NO memory-editing, NO search — deliberately out of scope. |
| `voice_tools.py` | Bridge from the app's LangChain tools to the Live protocol (which has NO auto tool-loop). `voice_function_declarations()` → `genai.types.FunctionDeclaration` list built from the SAME `agent_tools.build_tools()` (reuses `memory_update`+`search_conversations`; `adjust_word_score` excluded) PLUS a voice-only **`score_word`** (plain BLOCKING tool — 3.1-flash-live is blocking-only; classifies a spoken target word into the scoring_service event types and applies it → one scoring path, matches text). `VoiceToolExecutor` executes a tool call server-side, records it (for per-turn persistence + the live score animation), and `drain_records()` per turn. |
| `chat_service.py` | Core turn engine. `run_agent_turn(conversation, user_content?, provider?, model?, extra_instruction?)` — NO db arg; persists via `repo`. user_content=None ⇒ opener flow. `extra_instruction` (e.g. the opener directive) is sent as a HumanMessage, NOT appended to the system prompt — keeps `contents` non-empty so Gemini doesn't reject the opener with `contents are required.`. Loads messages via `repo.load_messages`. `_history_messages` rebuilds AIMessage(tool_calls)+ToolMessage pairs with ORIGINAL ids (providers reject mismatches). `_flatten_content` — Gemini returns content as block LIST; always flatten. Tool loop max 6 iterations; tool errors returned to model, never raised. `_maybe_set_title` after first exchange (failure swallowed). **Resolves the per-user model** via `model_service.resolve_for_user(conversation.user_id)` and passes its key+model into `get_chat_model` and the title `get_utility_model` (the explicit `provider`/`model_name` args still override if given). |
| `judge_service.py` | `judge_user_message(conversation_id, user_message_id, user_id=…)` — NO db arg. Judges ALL tracked words (via `repo.list_words`, not just conversation targets) with structured output (`JudgeResult`), last 6 messages as context, maps classifications to scoring events. Resolves the per-user model (`resolve_for_user(user_id)`) for the judge call. Any exception ⇒ returns [] (never breaks chat). |
| `scoring_service.py` | SINGLE SOURCE OF TRUTH for scores. `apply_event(word, event_type, ...)` — NO db arg; persists via `repo.save_word`/`repo.insert_event`. Matrix: perfect_unprompted +5, perfect_prompted +3, awkward +1, wrong −2, passive +0.5, manual=delta. NO daily cap — gains apply in full (a word can go 0→100 in one day); only clamp is 0–100. `apply_decay(word)` (lazy, −1/week after 14 idle days, called on words list reads). `pick_target_words(limit?, user_id=…)` (lowest score, then least recently used; excludes score 100). |
| `prompt_builder.py` | `build_system_prompt(conversation)` — NO db arg; reads memory files via `memory_service` (Mongo). Assembles CHAT_SYSTEM_TEMPLATE: persona block (from persona file, fallback if unset) → identity → memory → **the user's FULL practice vocabulary** (`_target_words_block`: ALL words with score<100 via `repo.list_words`, lowest-score-first, each `[id=…]`+score+register+memory-hook; the conversation's seeded `target_word_ids` get a ★ "focus first" hint but the persona may use ANY listed word — mastered words omitted so they're not drilled) → rich TIME block (`_time_block`) → category. `_part_of_day(hour)` → morning/midday/afternoon/evening/late night (no season/weather — deliberately not inferred, see prompts.py). |
| `memory_service.py` | Pure free-text markdown engine for identity/memory/persona. **identity & memory** are the SHARED human files (one `memory_files` doc each per user). **persona is PER-PERSONA**: the `persona` "file" transparently routes to the user's ACTIVE persona's `content` (in the `personas` collection), so callers still say `read_file("persona", uid)` unchanged. `_active_persona(uid)` resolves the active persona (self-heals: migrates a legacy single `persona` memory_files doc into a Persona row, or creates an empty one, and repairs a stale active pointer). `active_persona_id(uid)` (public — used to stamp/scope conversations). `_set_content` is the single write seam (persona→active persona doc, shared→memory_files). Same read-modify-write logic; NO ids/stamps. `normalize_file`, `append`, `edit(old,new,replace_all)`, `parse_lines`, `read_file`, `write_raw`, `reset_file`, `ensure_files` (bootstraps shared files + guarantees ≥1 persona). `build_persona_content(form, existing_raw="")` composes persona markdown preserving `## Relationship memories` (fields now include a `Voice:` line — the Gemini Live voice for voice mode, defaulted from Gender via `config.resolve_voice` when the form leaves it blank); `set_persona(form, uid, persona_id?)` targets a specific persona (edit/create) or the active one (onboarding); `parse_persona_fields(content)` extracts name/relation/gender/**voice**/personality/speaking_style for the persona cards. |
| `search_service.py` | `search(query, mode=bm25|regex, conversation_id?, n_before, n_after, full_conversation, max_results, exclude_conversation_id?, user_id=…, persona_id?)` — NO db arg; loads messages via `repo.all_messages` (which, given `persona_id`, limits to that persona's conversations). BM25 (rank_bm25) / regex ranking still done IN PYTHON (Mongo is only the store), context windows, overlapping-window dedup per conversation. `format_hits_for_llm` renders tool output text. NOTE: BM25 IDF is non-positive for a term present in EVERY document, so single-message corpora return no hits — a known quirk, not a bug. |
| `topic_service.py` | `suggest_topics(target_words, user_id=…, persona_id?)` (structured `TopicList`; uses identity+memory+`repo.recent_conversations` titles **scoped to the active persona**), `enrich_word(text, kind, user_id=…)` (structured `WordEnrichment`), `structure_onboarding_info(raw_about, persona_name, today, user_id=…)` (structured `OnboardingFacts{identity,memory,persona}`; `None` on LLM failure). ALL take `user_id` and resolve the per-user model (`resolve_for_user`) inside their try. All return empty/None on any LLM failure. |
| `persona_catalog.py` | Curated public-persona "Discover" catalog. Reads `app/assets/personas.json` (NOT the DB — identical for all users, saves free-tier Mongo) and caches via `lru_cache`. `categories()` (grouped list for the UI), `get_entry(catalog_id)` (one figure by stable id). Choosing one COPIES it into the user's `personas` (see `routers/personas.py` `/catalog/{id}/use`); the JSON is read-only. `avatar_url`s are filled manually in the JSON; `description` seeds the copy's Personality. |
| `agent_tools.py` | `build_tools(current_conversation_id, user_id=…, persona_id?)` → LangChain @tool list built per-request (closures over `user_id`+`persona_id`, NOT a db session): **`memory_update(file, action, text?, old_string?, new_string?, replace_all?)`** — single tool, `append`/`edit` (empty new_string deletes); constrained params are `Literal` enums so the model can't send junk — `file` ∈ identity|memory|persona (still defensively `normalize_file`'d), `action` ∈ append|edit; outputs use bare names, no `.md`; schema descriptions carry subject-free/date rules; `search_conversations` (excludes current conversation; **scoped to `persona_id`** so a companion only recalls chats it had); `adjust_word_score(word_id: str, delta, reason)` → manual event via `repo.get_word` + `scoring_service.apply_event`. Tool outputs stored on assistant Message.tool_calls (surfaced in UI Developer mode). |

Iron rules: services NEVER touch pymongo directly — all persistence goes through `app/repo.py` (the single swappable data layer). Every LLM call is scoped to the calling user's OWN key + tier — resolve via `model_service.resolve_for_user(user_id)` and pass its `provider/model/api_key` into the factory; NEVER call a factory with the app's own key. User keys are stored ENCRYPTED (`crypto_service`) — never log or return plaintext. LLM calls in judge/topics/enrichment/title/onboarding-structuring are best-effort (swallow failures). Never change scoring outside scoring_service.py — voice mode's `score_word` (in `voice_tools.py`) routes through it too, so voice + text scores stay identical (one matrix, no double-count). New agent tools go in agent_tools.py (shared by text + voice via the `voice_tools` bridge) + get mentioned in prompts.py tool rules. Voice mode reuses the whole text system prompt + `VOICE_MODE_INSTRUCTION`; keep voice logic in `voice_service.py`/`voice_tools.py` only. Memory files are pure free-text (no IDs/stamps) — dates belong in the text ONLY for time-bound facts, written by the agent itself, resolved as absolute dates against the TIME block. identity/memory are SHARED across a user's personas; the `persona` file is PER-PERSONA (routes to the active persona) — a user may keep several companions and switch; threads + conversation-search are persona-scoped, but words/scores/identity/memory are shared. `config.py`'s `user_timezone` (default Asia/Kolkata) drives all temporal computation.
