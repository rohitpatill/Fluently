## Auth (Google OAuth "Continue with Google") ‚Äî how a real user is resolved
Google-only login (no passwords). Server-side Authorization Code flow: Google tokens NEVER reach the browser. `deps.get_current_user` reads the signed JWT session cookie ‚Üí yields the internal `user_id`; every router injects it via `Depends(get_current_user)` INSTEAD of the old hardcoded `DEFAULT_USER_ID`. `repo`/services already accept `user_id`, so login = passing the real id through the same filters. First Google user to log in ADOPTS the legacy `"default"` data (one-time). Sessions are stateless (signed cookie, no sessions collection). Secrets/URLs are `.env`-only (see `config.py`) ‚Äî production is a config change, never a code change.

Covers words/events/conversations/messages/memory-files CRUD + dashboard aggregations + search loads + purge, plus **users/auth**: `get_user`, `get_user_by_sub`, `has_any_user`, `upsert_user_from_google` (find-or-create by Google `sub`, refreshes profile), `reassign_default_data` (one-time move of all `DEFAULT_USER_ID` docs to a new user). |
| `models.py` | PLAIN document classes (NOT ORM): `Conversation`, `Message`, `Word`, `WordEvent`, **`User`** ‚Äî each with `to_doc()`/`from_doc()` and a string `id` (ObjectId hex). Data docs carry `user_id`; a `User` doc's own `_id` IS that `user_id` (fields: `google_sub`, `email`, `name`, `picture`, `adopted_default`, `created_at` ‚Äî no passwords/tokens stored). | `deps.py` | FastAPI auth dependencies ‚Äî the SINGLE seam that resolves the current user. `get_current_user(request) -> user_id` (reads the session cookie ‚Üí `auth_service.decode_session_jwt` ‚Üí confirms the user exists; 401 otherwise). `get_current_user_obj(request) -> User` (same, returns the full profile for `/me`). Tests override these to run as a fixed user. |

## Data flow (one chat turn)
`routers/chat.py` ‚Üí `services/chat_service.run_agent_turn(conversation, user_content, ...)` (store user msg via `repo` ‚Üí `prompt_builder` system prompt ‚Üí rebuild history incl. tool calls ‚Üí bind_tools invoke loop ‚â§6 iters ‚Üí store assistant msg + tool_calls JSON ‚Üí auto-title) ‚Üí `services/judge_service.judge_user_message(conversation_id, user_message_id)` (structured output ‚Üí `scoring_service.apply_event`) ‚Üí response with scoring_events. No DB session is threaded through ‚Äî all persistence goes through `repo` (Mongo).

## Storage note
MongoDB Atlas (via `mongo.py` + `repo.py`); NO SQLite/SQLAlchemy anywhere. The 3 memory markdown files are documents in the `memory_files` collection (read-modify-write of the whole string), not on-disk files. Every data document carries `user_id` (a real Google user's id post-OAuth; the `"default"` sentinel only for un-adopted legacy data). Users live in the `users` collection.
