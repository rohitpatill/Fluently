## Auth (Google OAuth "Continue with Google") — how a real user is resolved
Google-only login (no passwords). Server-side Authorization Code flow: Google tokens NEVER reach the browser. `deps.get_current_user` reads the signed JWT session cookie → yields the internal `user_id`; every router injects it via `Depends(get_current_user)` INSTEAD of the old hardcoded `DEFAULT_USER_ID`. `repo`/services already accept `user_id`, so login = passing the real id through the same filters. First Google user to log in ADOPTS the legacy `"default"` data (one-time). Sessions are stateless (signed cookie, no sessions collection). Secrets/URLs are `.env`-only (see `config.py`) — production is a config change, never a code change.

## Data flow (one chat turn)
`routers/chat.py` (gated by `require_model_configured`) → `services/chat_service.run_agent_turn(conversation, user_content, ...)` (store user msg via `repo` → **resolve the user's model+key via `model_service.resolve_for_user`** → `prompt_builder` system prompt → rebuild history incl. tool calls → bind_tools invoke loop ≤6 iters → store assistant msg + tool_calls JSON → auto-title) → `services/judge_service.judge_user_message(conversation_id, user_message_id)` (resolves the same per-user model → structured output → `scoring_service.apply_event`) → response with scoring_events. No DB session is threaded through — all persistence goes through `repo` (Mongo).

## Bring-your-own-key model config
Each user supplies their OWN Gemini key at onboarding ("How smart should I be?") + picks a tier (**Swift**=gemini-3.1-flash-lite / **Sage**=gemini-3.5-flash, defined in `config.MODEL_TIERS`). The chosen tier governs EVERY LLM call for that user (chat + judge + utility). The key is stored ENCRYPTED (`services/crypto_service.py`, Fernet, `ENCRYPTION_KEY` from `.env`) on the `User` doc; `services/model_service.resolve_for_user` decrypts it per request. The app's own `.env` provider keys are NO LONGER used for real users. Managed via `routers/model.py` (+ Settings Brain card / onboarding step 3).

## Multi-persona
A user may keep several **personas** (companions) and switch the active one (`User.active_persona_id`). `identity` + `memory` (the human's facts/life) are SHARED across all personas; each **persona owns its own markdown** (identity fields + its own `## Relationship memories`) in the `personas` collection, plus an optional public-URL avatar. Conversations carry a `persona_id` and are **persona-scoped** (list + agent conversation-search + opener/topic context all filter to the active persona); words/scores stay global. `memory_service` routes the `persona` "file" to the active persona transparently. Deleting a persona cascades its conversations (keeps words); the last one can't be deleted. Curated **Discover** catalog in `assets/personas.json` → `routers/personas.py` copies a chosen figure into the user's personas. Migration for pre-multipersona users: `backend/migrate_personas.py` (dry-run default, `--commit`) turns each user's single persona into one active Persona + backfills conversation `persona_id`s (also self-heals lazily on first read).

## Voice mode (real-time audio)
The user can TALK to the active persona (mic button in the chat composer). A WebSocket (`routers/voice.py`) proxies duplex audio between the browser and **Gemini Live** (`config.voice_model`), paid by the user's OWN key. It reuses everything text chat has — the same dynamic system prompt (persona/identity/memory/full word list/time), the same tools (`memory_update`/`search_conversations` via the `voice_tools` bridge) — plus `prompts.VOICE_MODE_INSTRUCTION` appended and a voice-only **`score_word`** tool. **Scoring differs by mode:** text scores via the post-turn `judge_service`; voice scores INLINE via `score_word` (the live model calls it per spoken target word) — both route through `scoring_service`, so numbers match. Each completed voice turn is persisted as normal user+assistant `Message`s, so the voice conversation appears in the chat thread afterward. Each persona has a `Voice:` (from `config.VOICES`), defaulted from its Gender.

## Fluently in-app voice assistant
A SEPARATE voice feature from persona chat: the APP itself talks to the user (help + a few
hands-free actions). Own module set (`services/assistant_service.py` + `services/assistant_tools.py`
+ `routers/assistant.py`, prompt `ASSISTANT_SYSTEM_TEMPLATE`), reusing ONLY the Gemini Live
transport + per-user key. Ephemeral: nothing persisted, nothing scored, not tied to a persona.
Loads identity+memory so it sounds personal; speaks first by name; STAY-IN-SCOPE prompt limits it
to explaining features + FOUR tools (`get_my_status`/`create_persona`/`add_word`/`switch_model_tier`,
each wrapping the existing code path). Fixed friendly voice (`Puck`), not the persona's. Frontend
opens it from a draggable logo FAB; successful actions invalidate the affected caches so screens
refresh live.

## Storage note
MongoDB Atlas (via `mongo.py` + `repo.py`); NO SQLite/SQLAlchemy anywhere. `identity`+`memory` are documents in the `memory_files` collection (read-modify-write of the whole string); each persona's markdown lives in the `personas` collection. Every data document carries `user_id` (a real Google user's id post-OAuth; the `"default"` sentinel only for un-adopted legacy data). Users live in the `users` collection.
